## Working Rules (항상 준수)

### 작업 시작 전
1. 어떤 작업을 할 것인지 명확히 설명한다.
2. 어떤 순서로 진행할지 계획을 제시한다.
3. **작업 계획을 세울 때 궁금한 사항은 반드시 사용자에게 확인한다.** 요구사항이 모호하거나 여러 해석이 가능하면 먼저 질문하고, 확인 후 진행한다.
4. `docs/작업/TODO.md`를 확인하여 현재 진행 상황을 파악한다.

### 작업 완료 후
1. 어떤 작업이 진행되었는지 리뷰 요약을 제공한다.
2. **`docs/작업/TODO.md`에서 완료된 항목을 즉시 `[x]`로 체크한다.**
3. 다음에 진행할 작업을 안내한다.
4. **git commit/push는 사용자가 명시적으로 요청할 때만 수행한다.** 작업 완료 후 자동으로 commit하지 않는다.

### 문서 동기화 규칙 (반드시 준수)
- **설계와 달라진 경우**: 코드 작성(추가/수정 등) 완료 후 설계 내용과 달라지거나 추가·삭제된 사항이 생기면
  해당 `docs/설계/` 문서를 반드시 **즉시** 업데이트한다.
- **TODO 동기화**: 아래 모든 상황에서 `docs/작업/TODO.md`를 **즉시** 업데이트한다.
  - 새 작업 항목이 추가될 때
  - 구현 내용이 변경/추가/삭제될 때
  - 작업이 완료될 때 → `[x]` 체크
  - 작업이 재작업/롤백될 때 → `[x]` 해제 후 내용 수정
  - 작업 진행 상태가 변경될 때 (예정 → 진행중 → 완료 → 재작업)
- **문서 위치 규칙**:
  - `docs/설계/` ? 설계 명세 문서 (PRD, FSD, 아키텍처, 데이터모델 등)
  - `docs/작업/TODO.md` ? **살아있는** 작업 추적 (현재 진행 + 다음 작업만 유지)
  - `docs/작업/완료_이력.md` ? 완료된 Phase 작업 + 핫픽스 이력 보존
  - 작업 완료 시 → TODO.md에서 `[x]` 체크 → 단락이 더 이상 진행 중이 아니면 완료_이력.md 하단에 시간순 추가 + TODO.md에서 제거 (TODO 비대화 방지)

### 코드 작성
- 주석은 상세하게 작성한다. 로직의 의도, 파라미터 의미, 주요 흐름을 설명한다.
- **모듈화**: 컴포넌트·ViewModel·서비스 등 최소 단위 객체는 재사용 가능하게 분리한다. 동일 로직이 두 곳 이상에 등장하면 공통 클래스/메서드로 추출한다.
- **상수화**: 고정 값(문자열 키, 숫자 기본값, 색상 코드 등)은 `const` 또는 `static readonly` 상수로 선언하여 한 곳에서 관리한다. 값 변경 시 상수 하나만 수정하면 전 계층에 일괄 반영되도록 한다.
- **변경 최소화 설계**: 향후 수정을 대비해 변경 영향 범위를 좁힌다. 인터페이스/추상화 활용, 하드코딩 분기 대신 데이터 주도 방식을 우선한다.
